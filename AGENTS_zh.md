# AGENTS.md 参考模板（中文）

> 面向 Codex、Codex CLI、以及其他支持 `AGENTS.md` 的 coding agent。

## 宗旨

本文件的目标不是让 agent 更有创造力，而是让它更稳定地交付可验证、可审查、可回滚的代码改动。

## 优先级（从高到低）

1. 可验证优先  
   先定义完成标准，再写代码。优先使用测试、lint、typecheck、构建、截图对比、复现步骤等可执行验证。

2. 先探索，再计划，再编码  
   中等及以上复杂度任务先读代码、找边界、确认依赖，再动手修改，避免“修错问题”。

3. 指令清晰、结构化、无冲突  
   任务描述应尽量像一条高质量 issue：背景、目标、约束、影响范围、验收条件、参考实现。

4. 小任务优先，增量推进  
   把大任务拆成若干个可独立验证的小步骤。每一步都应能被确认对错。

5. 最小改动，尊重现有模式  
   优先复用现有实现、命名、测试习惯和目录结构。不要为了“更优雅”而偏离仓库既有模式。

6. 先保护行为，再清理结构  
   做 cleanup、refactor、deslop 时，先用回归测试或其他证据锁住行为，再做结构整理。

7. 保持上下文干净  
   只保留与当前任务直接相关的上下文。避免把长历史、无关文件、重复说明塞进工作上下文。

8. 默认单 agent，必要时再并行  
   先用单 agent 跑通闭环；只有在任务天然可拆、并行收益明显时才启用多 agent。

9. 写作与审查分离  
   条件允许时，让一个 agent 实现，让另一个 agent 做独立审查或验证。

10. 高风险操作必须加护栏  
   对破坏性、不可逆、涉密或高成本操作，要显式收紧权限、确认风险、记录依据。

## 默认执行协议

1. 探索  
   先阅读相关代码、测试、配置和文档，确认真正的变更面。

2. 计划  
   用简短步骤说明准备怎么做、如何验证、可能的风险点。

3. 实现  
   优先最小可行修改。除非任务明确要求，否则不要顺手重构无关代码。

4. 验证  
   运行与改动匹配的验证。小改动做轻量验证；跨模块改动做完整验证。

5. 报告  
   交代改了什么、如何验证、还剩什么风险或未覆盖项。

## 具体约束

- 不要为未来可能需要的场景提前加入复杂抽象、扩展点或新依赖。
- 不要改动与任务无关的文件、格式、命名或注释。
- 不要删除自己没有充分理解的逻辑；先解释其作用，再决定是否处理。
- 不要把“看起来更漂亮”当作变更理由；优先保证正确、稳定、可审查。
- 如现有行为没有保护，先补回归测试，再进行整理。
- 优先复用已有工具函数、测试夹具、错误处理模式和配置约定。

## 推荐的任务输入格式

最好提供以下信息：

- 目标：要解决什么问题
- 范围：允许改哪些目录或模块
- 约束：不能引入什么、副作用边界是什么
- 验收：什么结果算完成
- 参考：已有相似实现、文档或 issue

## 推荐的交付格式

- 结果：本次改动完成了什么
- 验证：运行了哪些检查，结果如何
- 风险：还有哪些未覆盖、待观察或需要人工确认的点

## 一句版原则

让 agent 在清晰边界内，小步、最小改动、可验证地完成任务，而不是在模糊目标下“聪明地写很多代码”。
