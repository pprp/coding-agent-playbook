# CLAUDE.md 参考模板（中文）

> 面向 Claude Code 的仓库级指导文档。

## 核心目标

让 Claude Code 在这个仓库里稳定地产出小步、可验证、符合现有风格的改动，而不是在上下文不清时直接跳到实现。

## 优先级（从高到低）

1. 给 Claude 一个验证闭环  
   每个任务都应尽量附带测试、命令、期望输出、截图标准或复现步骤，让 Claude 能自己检查结果。

2. 先 Explore，再 Plan，再 Code  
   中大型任务先进入计划式工作流，分离“理解问题”和“修改代码”。

3. Prompt 要具体，最好像 issue  
   提供明确目标、目录范围、不能碰的边界、验收条件，以及可参考的现有实现。

4. 一次只做清晰的小任务  
   Claude 更适合处理边界明确、几百行以内、可独立验证的任务，而不是模糊的大工程。

5. 遵循仓库现有模式  
   优先模仿现有代码、测试、命名和错误处理方式，不要为了统一审美而重写周边实现。

6. 清理和行为变更尽量分开  
   如果既要修逻辑又要整理结构，先锁行为，再做结构清理；不要把两类改动混成一个模糊大 diff。

7. 控制上下文密度  
   只给当前任务真正相关的文件、日志和说明。上下文过长、过杂会降低稳定性。

8. 先单会话，必要时再并行  
   默认一个 Claude 会话完成主线工作；只有在 review、测试、迁移拆分等天然可并行场景下才启用多会话。

9. 并行时保持角色分工  
   写代码、写测试、做 review 最好分开，避免同一上下文既当作者又当审稿人。

10. 对高风险操作收紧权限  
   涉及删除、重写、发布、密钥、远程操作或高成本动作时，要明确收紧权限和检查点。

## Claude Code 默认工作流

1. 探索阶段  
   阅读相关文件、测试、配置、报错和文档，确认要改哪里、不要改哪里。

2. 计划阶段  
   写出简短步骤：准备改什么、怎么验证、哪些点可能有风险。

3. 实现阶段  
   先做最小变更，再决定是否需要扩展。除非任务明确要求，否则不要顺手重构无关代码。

4. 验证阶段  
   运行与变更相匹配的验证，如测试、lint、typecheck、build、截图或手工复现。

5. 汇报阶段  
   说明改动结果、验证证据、剩余风险，以及是否还需要人工判断。

## 对 Claude 的具体要求

- 中大型任务优先使用计划式流程，不要直接编码。
- 任务不清晰时先指出歧义，再继续推进；但对低风险、可逆的下一步可以直接执行。
- 不要为了“未来可能有用”提前加入抽象层、配置项或新依赖。
- 不要顺手修改无关文件、注释、格式或命名。
- 如需 cleanup / refactor，先补保护行为的测试或其他证据。
- 尽量把说明写短，把验证做实。

## 建议放进 Prompt 的信息

- 背景：为什么要改
- 目标：想要的结果
- 范围：允许改哪些地方
- 禁区：不要碰什么
- 验收：如何判断完成
- 参考：类似模块、issue、文档、截图

## 建议的最终回复格式

- 结果：做成了什么
- 验证：跑了什么，结果如何
- 风险：哪些点还需要观察或人工确认

## 一句版原则

不要让 Claude 在模糊指令下直接写很多代码；要让它在清晰约束里，通过探索、计划、最小改动和验证闭环稳步完成任务。
